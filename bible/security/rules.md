## Rule SEC-AUTH-BRUTE-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing any authentication endpoint (login, password verify, MFA verify, API token issuance).

### Statement
Failed authentication attempts must be rate-limited per account and per source IP. Acceptable mechanisms: progressive lockout (lock for N minutes after K fails), exponential backoff, CAPTCHA after threshold. The limit applies even when the username does not exist.

### Violation
```python
@app.route('/login', methods=['POST'])
def login():
    user = User.query.filter_by(email=email).first()
    if user and user.check_password(password):
        return success()
    return error()  # unlimited retries
```

### Pass
```python
@app.route('/login', methods=['POST'])
@limiter.limit('5 per minute', key_func=lambda: request.json['email'])
@limiter.limit('20 per minute', key_func=get_remote_address)
def login():
    ...
```

### Enforcement
Code review. Look for auth endpoints without rate-limit decorators or middleware. Frameworks: Flask-Limiter, django-ratelimit, express-rate-limit.

### Rationale
Credential stuffing is automated and cheap. Without a per-account limit, an attacker tries a billion known-password pairs in hours. Per-IP plus per-account limits are the structural defense.

## Rule SEC-AUTH-ENUM-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing login, password reset, signup, or any endpoint that distinguishes between 'this account exists' and 'this account does not exist'.

### Statement
Login and password-reset endpoints must return identical responses (same status, same body, same timing) for valid and invalid accounts. Discoverability of registered usernames or emails is a violation.

### Violation
```python
if not user:
    return 'No such email', 404
if not user.check_password(password):
    return 'Wrong password', 401
```

### Pass
```python
if not user or not user.check_password(password):
    return 'Invalid email or password', 401
```

### Enforcement
Code review. Look for branched error messages or distinct status codes on auth failure paths.

### Rationale
Account enumeration powers targeted phishing, credential stuffing prioritization, and abuse reporting workflows. Uniform responses remove the oracle.

<!-- RULE START: SEC-AUTH-HASH-001 -->
## Rule SEC-AUTH-HASH-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When persisting user passwords, password-equivalent secrets, or any credential that must withstand offline cracking.

### Statement
Passwords must be hashed with bcrypt, argon2, or scrypt. Single-pass MD5, SHA-1, SHA-256, or SHA-512 are never acceptable for password storage. The work factor must match current industry guidance (bcrypt cost >= 12, argon2id memory >= 64 MiB).

### Violation
```python
import hashlib
hashed = hashlib.sha256(password.encode()).hexdigest()
user.password = hashed
```

### Pass
```python
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))
user.password = hashed.decode()
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_auth_authz: regex flags hashlib.md5/sha1/sha256/sha512 (Python), md5()/sha1() (PHP), crypto.createHash('md5'|'sha1'|'sha256') (Node) in proximity to the identifier 'password'/'passwd'/'pwd' or assignment to a *.password field.

### Rationale
Fast hashes are designed for throughput; password hashes must be slow on purpose. GPU rigs crack billions of SHA-256 per second; bcrypt at cost 12 caps adversary throughput by roughly six orders of magnitude. The choice of algorithm is structural, not optional.

<!-- RULE END: SEC-AUTH-HASH-001 -->

## Rule SEC-AUTH-HASH-002

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When generating or storing the salt that accompanies a password hash.

### Statement
Salts must be unique per user and generated by the hashing library's gensalt helper. Never reuse a global salt, derive the salt from the username, or hand-roll salt generation from a non-CSPRNG.

### Violation
```python
SALT = b"my-app-salt"
hashed = bcrypt.hashpw(password.encode() + SALT, bcrypt.gensalt())
```

### Pass
```python
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
# bcrypt embeds the per-call random salt in its output string.
```

### Enforcement
Code review. Static checks for module-level salt constants or salt = '<literal>' patterns help.

### Rationale
Per-user salts defeat rainbow tables and force the attacker to mount a separate attack per credential. Library-generated salts come from a CSPRNG; hand-rolled salts are routinely weak.

## Rule SEC-AUTH-LOGOUT-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing logout for any session-bearing authentication system.

### Statement
Logout must invalidate the server-side session: delete the session row, revoke the JWT (if a denylist is maintained), or rotate the session key. Clearing the cookie client-side without server-side invalidation is a violation.

### Violation
```python
@app.route('/logout')
def logout():
    response = redirect('/')
    response.delete_cookie('sessionid')
    return response
```

### Pass
```python
@app.route('/logout')
def logout():
    session_store.delete(session['sid'])  # server-side invalidation
    response = redirect('/')
    response.delete_cookie('sessionid')
    return response
```

### Enforcement
Code review.

### Rationale
A cookie-only logout leaves the session valid on any other device or in any captured copy of the cookie. Server-side invalidation makes the credential genuinely unusable.

<!-- RULE START: SEC-AUTH-MFA-001 -->
## Rule SEC-AUTH-MFA-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing TOTP-based multi-factor authentication.

### Statement
MFA shared secrets must be encrypted at rest (database-level or field-level), never stored in plaintext. TOTP verification accepts only the current 30-second period plus exactly one adjacent period for clock skew. Wider windows are violations.

### Violation
```python
user.totp_secret = generate_secret()
user.save()  # plaintext column
if pyotp.TOTP(secret).verify(code, valid_window=10):
    pass
```

### Pass
```python
user.totp_secret = encrypt_at_rest(generate_secret())
user.save()
if pyotp.TOTP(secret).verify(code, valid_window=1):
    pass
```

### Enforcement
Schema review (encrypted column type), code review (valid_window argument).

### Rationale
A wide TOTP window expands the brute-force keyspace; plaintext secrets give a database leak full account takeover. Both undermine the second factor.

<!-- RULE END: SEC-AUTH-MFA-001 -->

## Rule SEC-AUTH-RESET-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing password reset, email change confirmation, or any single-use token-bearing link sent to a user.

### Statement
Reset tokens must be single-use (invalidated on first successful use), time-limited to one hour or less, and invalidated when the underlying credential changes by any other path. Tokens are CSPRNG-generated per SEC-AUTH-TOKEN-001.

### Violation
```python
user.reset_token = secrets.token_urlsafe(32)
user.save()
# No expiry field; token reusable forever.
```

### Pass
```python
user.reset_token = secrets.token_urlsafe(32)
user.reset_token_expires_at = datetime.utcnow() + timedelta(minutes=30)
user.save()
# Verify side: check token, expiry, mark token consumed.
user.reset_token = None
user.reset_token_expires_at = None
user.save()
```

### Enforcement
Schema review (expiry column, consumed flag), code review (consumption logic).

### Rationale
A reset token without expiry is a permanent backdoor that survives token exposure (email leak, browser history). A non-single-use token can be replayed after the user thinks they have rotated credentials.

<!-- RULE START: SEC-AUTH-TIMING-001 -->
## Rule SEC-AUTH-TIMING-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When comparing two byte/string values where one side is attacker-influenced and the other is a secret (session token, API key, password hash, HMAC tag, reset token).

### Statement
Secret comparisons must use constant-time helpers: Python `hmac.compare_digest`, Node `crypto.timingSafeEqual`, PHP `hash_equals`, Java `MessageDigest.isEqual`. Equality operators (`==`, `===`) leak timing information.

### Violation
```python
if user_supplied_token == stored_token:
    grant_access()
```

### Pass
```python
import hmac
if hmac.compare_digest(user_supplied_token.encode(), stored_token.encode()):
    grant_access()
```

### Enforcement
Code review. Grep for ` == ` or ` === ` adjacent to identifiers containing 'token', 'secret', 'hmac', 'signature', 'mac', 'tag'.

### Rationale
Timing attacks let a remote attacker peel a secret one byte at a time by observing micro-latency. Constant-time comparison closes the side channel structurally.

<!-- RULE START: SEC-AUTH-TOKEN-001 -->
## Rule SEC-AUTH-TOKEN-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When generating session tokens, password-reset tokens, email-verification tokens, API keys, OAuth state values, CSRF tokens, or any other security-sensitive random value.

### Statement
Security tokens must be produced by a cryptographically secure pseudo-random number generator: Python `secrets`, Node `crypto.randomBytes`, PHP `random_bytes`, Java `SecureRandom`. The token must be at least 128 bits of entropy (16 random bytes hex-encoded or 22 base64url characters).

### Violation
```python
import random, string
token = ''.join(random.choices(string.ascii_letters + string.digits, k=20))
```

### Pass
```python
import secrets
token = secrets.token_urlsafe(32)  # 256 bits of entropy
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_auth_authz: regex flags Math.random(), random.random(), random.choices(), random.randint(), random.choice(), rand(), mt_rand(), srand() within proximity to identifiers 'token', 'session', 'secret', 'csrf', 'key', 'nonce', 'reset', 'verify', 'api_key', 'sessid', 'salt'.

### Rationale
`random.random` is seeded by predictable system state and produces predictable streams. Once an attacker recovers a few outputs, they can predict future tokens. CSPRNGs source entropy from the kernel and have no recoverable state.

## Rule SEC-AUTH-TOKEN-002

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When issuing session tokens (cookies, Authorization headers) from the application server.

### Statement
Session cookies must be set with Secure (HTTPS only), HttpOnly (no JS access), and SameSite=Lax or Strict. Authorization-header bearer tokens are transmitted only to HTTPS endpoints. Plain HTTP transmission of any session credential is forbidden.

### Violation
```python
response.set_cookie('sessionid', token)
```

### Pass
```python
response.set_cookie(
    'sessionid', token,
    secure=True, httponly=True, samesite='Lax',
    max_age=86400,
)
```

### Enforcement
Framework config review (Django SESSION_COOKIE_SECURE/HTTPONLY/SAMESITE, Flask SESSION_COOKIE_SECURE, Express express-session cookie.secure/httpOnly/sameSite). Static analysis flags `set_cookie` without these flags.

### Rationale
Secure stops MITM credential theft on shared networks. HttpOnly stops XSS-driven cookie exfiltration. SameSite stops cross-site CSRF and some token-leak vectors. All three are zero-cost browser-level defenses.

## Rule SEC-AUTHZ-DEFAULT-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When adding a new endpoint, GraphQL field, server action, or admin tool to the application.

### Statement
New endpoints default to deny: they must opt in to a permission policy. The framework or application middleware must reject requests to endpoints without an explicit policy. 'No decorator means public' is a violation.

```python
# Framework defaults to allow; new endpoint inherits public access.
@app.route('/internal/secret-debug')
```

```python
# App-level middleware rejects any handler without @policy(...) marker.
@app.route('/internal/secret-debug')
@policy('admin', reauth=True)
```

### Rationale
Deny-by-default is the only configuration that fails safe when a developer forgets to think about authorization. Inverting the default turns absence-of-thought into a security finding instead of a silent leak.

Code review of route registration, middleware coverage.

## Rule SEC-AUTHZ-IDOR-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When loading or mutating any record where the record ID is supplied by the request (URL path param, query string, body field).

### Statement
Object access must be verified against the authenticated user's permissions, not just the existence of the record. Loading `Order.find(params[:id])` without checking `order.user_id == current_user.id` (or equivalent permission predicate) is a violation. This applies to read, update, delete, and side-channel exposure (filenames, redirect targets).

### Violation
```python
@app.route('/orders/<int:id>')
@login_required
def view_order(id):
    order = Order.query.get_or_404(id)
    return render(order)
# Any logged-in user can read any order by guessing IDs.
```

### Pass
```python
@app.route('/orders/<int:id>')
@login_required
def view_order(id):
    order = Order.query.filter_by(id=id, user_id=current_user.id).first_or_404()
    return render(order)
```

### Enforcement
Code review. Look for `.get(pk=...)`, `Model.find(...)`, `.objects.get(id=...)` where the ID source is request data and no follow-up ownership/permission check is present.

### Rationale
Insecure direct object reference is one of the OWASP Top 10 staples because record-by-ID is the default ORM pattern and the permission check is easy to omit. Every loader of user-influenced IDs must also gate access.

## Rule SEC-AUTHZ-MASS-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When constructing or updating a model instance from a request body (REST POST/PATCH, GraphQL mutation, form submission).

### Statement
Model fields populated from user input must use an explicit allowlist (Pydantic schema with declared fields, Rails strong parameters `permit(...)`, Django ModelForm `fields = [...]`, ActiveRecord `params.require(...).permit(...)`). Passing the raw request body to a constructor or update call is a violation.

### Violation
```python
# Python
user = User(**request.json)  # role, is_admin, balance all accepted from caller
user.save()
```
```ruby
# Rails
user.update_attributes(params[:user])
```
```javascript
// Node
Object.assign(user, req.body); user.save();
```

### Pass
```python
class UserCreate(BaseModel):
    email: EmailStr
    name: str

user = User(**UserCreate(**request.json).dict())
user.save()
```
```ruby
user.update_attributes(params.require(:user).permit(:email, :name))
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_auth_authz: regex flags `Model(**request.json|request.body|req.body)`, `Object.assign(model, req.body)`, `update_attributes(params[`, `Model.create(**request.`. The detector encodes the common mass-assignment idioms across Python, Ruby, and Node.

### Rationale
Mass assignment lets the caller set fields the developer never meant to be writable: role, is_admin, balance, password_hash. The structural defense is an explicit allowlist at the request-binding layer, not field-by-field deny logic.

**Mandatory**: false

### Trigger
When the application has roles, permissions, or capabilities that gate behavior.

```python
def list_users(actor: User):
```

## Rule SEC-AUTHZ-TENANT-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: false

### Trigger
When the application has multiple tenants (organizations, workspaces, accounts) sharing the same database or storage.

### Statement
Multi-tenant queries must filter by tenant ID at the data layer (repository method, ORM scope, query helper), not only at the API layer. Tenant scoping enforced only in controllers leaks data through internal callers, background jobs, or any code path that bypasses the controller.

```python
# Controller-only filter:
@app.route('/orders')
@login_required

def rebuild_index():
```

```python
class TenantQuery(Query):

Order.query.with_tenant(current_user.org_id).all()
# Repository wrappers enforce the filter at the data layer.
```

### Enforcement
Code review of repositories and ORM scopes. Static analysis can flag raw `Model.query.all()` outside of tenant-scoped wrappers.

### Rationale
Cross-tenant data leaks are the highest-impact authorization bug in SaaS systems: a single missing filter exposes every customer at once. Data-layer enforcement is the structural defense.

<!-- RULE START: SEC-CRYPTO-ALGO-001 -->
## Rule SEC-CRYPTO-ALGO-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing symmetric encryption for data at rest, message authentication, or session-token sealing.

### Statement
Symmetric encryption must use AES-256-GCM, ChaCha20-Poly1305, or AES-256-SIV. ECB mode is forbidden. DES, 3DES, RC4, Blowfish are forbidden. Modes without authentication (CBC, CTR alone) require an explicit HMAC.

### Violation
```python
from Crypto.Cipher import AES
cipher = AES.new(key, AES.MODE_ECB)  # ECB leaks plaintext patterns
ct = cipher.encrypt(plaintext)
```

### Pass
```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
aesgcm = AESGCM(key)
nonce = os.urandom(12)
ct = aesgcm.encrypt(nonce, plaintext, associated_data=None)
```

### Enforcement
Code review. Static analysis flags AES.MODE_ECB, AES.MODE_CBC without an HMAC, DES, 3DES, RC4 usage.

### Rationale
ECB mode preserves plaintext patterns and is famously broken (the 'ECB penguin'). Unauthenticated modes allow ciphertext modification. AEAD modes are the structural defense; they bundle confidentiality and integrity.

<!-- RULE END: SEC-CRYPTO-ALGO-001 -->

<!-- RULE START: SEC-CRYPTO-ALGO-002 -->
## Rule SEC-CRYPTO-ALGO-002

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When generating asymmetric key pairs (RSA, EC, Ed25519) for signing, encryption, or TLS.

### Statement
RSA keys are minimum 2048 bits; new keys should be 3072 or 4096 bits. EC keys use P-256 or stronger, or Ed25519. RSA-1024, DSA-1024, and short-curve EC keys are forbidden.

### Violation
```python
rsa.generate_private_key(public_exponent=65537, key_size=1024)
```

### Pass
```python
rsa.generate_private_key(public_exponent=65537, key_size=3072)
# or
ed25519.Ed25519PrivateKey.generate()
```

<!-- RULE START: SEC-CRYPTO-IV-001 -->
## Rule SEC-CRYPTO-IV-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When performing symmetric encryption that requires an IV or nonce (AES-GCM, AES-CBC, ChaCha20-Poly1305).

### Statement
IVs and nonces must be unique per encryption operation, generated by a CSPRNG, never reused with the same key. For GCM, nonce reuse with the same key is catastrophic (allows full plaintext recovery of both messages).

### Violation
```python
IV = b'\x00' * 12  # global constant; reused for every message
aesgcm.encrypt(IV, plaintext, None)
```

### Pass
```python
import os
nonce = os.urandom(12)
ciphertext = aesgcm.encrypt(nonce, plaintext, None)
# Store nonce alongside ciphertext.
```

### Enforcement
Code review. Look for module-level IV/nonce constants or fixed-byte assignments.

### Rationale
Nonce reuse in GCM is one of the most catastrophic crypto failures: an attacker who sees two GCM messages with the same key+nonce can recover plaintexts and forge new authenticated messages. Per-message randomness is non-negotiable.

## Rule SEC-CRYPTO-TLS-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When configuring TLS servers, HTTPS clients, or any code that establishes encrypted transport.

### Statement
TLS 1.2 is the minimum; TLS 1.0 and 1.1 are disabled. The cipher suite excludes RC4, DES, 3DES, MD5-based MACs, and export-grade ciphers. Server configuration explicitly sets the floor.

```python
ctx = ssl.SSLContext(ssl.PROTOCOL_SSLv23)  # accepts SSL 3.0 / TLS 1.0
```

```python
ctx = ssl.SSLContext(ssl.PROTOCOL_TLS)
```

### Enforcement
Server config review (nginx ssl_protocols, Apache SSLProtocol, Go tls.Config MinVersion, Java sslContext.getProtocol). Mozilla SSL Configuration Generator as reference.

### Rationale
TLS 1.0 and 1.1 carry known weaknesses (BEAST, POODLE, weak MACs). Modern servers and clients all support 1.2+. Raising the floor is configuration only -- no application impact.

```python
@app.route('/redirect')
```

### Rationale

<!-- RULE START: SEC-INJ-PATH-001 -->
## Rule SEC-INJ-PATH-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: false

### Trigger
When file paths are constructed from user input (uploads, downloads, includes, template loading, archive extraction).

### Statement
User-supplied path components must be validated against path traversal: reject `..`, null bytes, and absolute paths; resolve to a canonical path; verify the result is within the intended base directory. The base directory check uses canonical paths on both sides.

### Violation
```python
def serve(filename):
    with open(f"/var/uploads/{filename}") as f:
        return f.read()
# filename='../../etc/passwd' escapes the base
```

### Pass
```python
import os
BASE = os.path.realpath("/var/uploads")
def serve(filename):
    candidate = os.path.realpath(os.path.join(BASE, filename))
    if not candidate.startswith(BASE + os.sep):
        raise PermissionError("path traversal")
    with open(candidate) as f:
        return f.read()
```

### Enforcement
Code review. Static analysis can flag `os.path.join` followed by `open(` with non-constant first argument.

### Rationale
Path traversal is the simplest form of arbitrary file disclosure, and it consistently appears in real incidents. The realpath + prefix-check pattern is the only structurally safe form; everything else is filtering that misses cases (URL-encoded `..`, double encoding, Windows path separators).

## Rule SEC-INJ-REDIR-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When an endpoint redirects the user to a URL supplied or influenced by request parameters (login-success redirects, return-to flows, deeplink handlers).

### Statement
Redirect destinations from user input must be validated against an allowlist of internal paths or an explicit list of permitted external hosts. Open redirect is a violation even when the receiving endpoint is otherwise harmless.

### Violation
```python
@app.route('/login')
def login():
    next_url = request.GET.get("next", "/")
    return redirect(next_url)
# next=//evil.example.com/phish is an open redirect
```

```python
from urllib.parse import urlparse
ALLOWED_HOSTS = {"", "app.example.com"}  # empty string == relative path
@app.route('/login')
def login():
    next_url = request.GET.get("next", "/")

    if parsed.netloc not in ALLOWED_HOSTS:
        next_url = "/"
    return redirect(next_url)
```

### Enforcement
Code review. Framework-specific redirect helpers (Django's `url_has_allowed_host_and_scheme`) provide a vetted check.

### Rationale
Open redirect is a phishing accelerator: attackers send links that look like they go to your domain and bounce the user to a credential-harvesting site. The allowlist pattern prevents that.

## Rule SEC-INJ-SQL-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_injection

### Trigger
When writing SQL strings that include any variable, user input, or function return value.

### Statement
Parameterized queries only. SQL must never be built by string concatenation, format, f-string, or interpolation of any value that is not a literal constant. Every dynamic value enters through a bind parameter.

### Violation
```python
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")
# or
query = "SELECT * FROM users WHERE email = '" + email + "'"
cursor.execute(query)
```

### Pass
```python
cursor.execute(
    "SELECT * FROM users WHERE id = :user_id AND status = :status",
    {"user_id": user_id, "status": status},
)
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_injection: regex scan flags f-strings, .format(), and `+`-concatenation in proximity to execute/cursor/raw-query call sites. PHPStan + PHPCS catch the equivalent PHP patterns.

### Rationale
SQL injection remains the single most common high-severity web vulnerability. Parameterization is the only structurally safe pattern: it removes the user-input-as-code attack surface entirely instead of trying to sanitize it.

<!-- RULE END: SEC-INJ-SQL-001 -->

<!-- RULE START: SEC-INJ-SQL-002 -->
## Rule SEC-INJ-SQL-002

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: false

### Trigger
When using ORM raw-query escape hatches (Django `raw()`, SQLAlchemy `text()`, Doctrine `getNativeQuery`, Eloquent `DB::raw`).

### Statement
ORM raw-query methods must use the ORM's parameter binding, never string interpolation. Escape hatches bypass the ORM's automatic parameterization; bound parameters bring it back.

### Violation
```python
# Django
User.objects.raw(f"SELECT * FROM users WHERE id = {user_id}")
# SQLAlchemy
session.execute(text(f"SELECT * FROM users WHERE id = {user_id}"))
```

### Pass
```python
# Django
User.objects.raw("SELECT * FROM users WHERE id = %s", [user_id])
# SQLAlchemy
session.execute(text("SELECT * FROM users WHERE id = :id"), {"id": user_id})
```

### Enforcement
Code review. Static analysis (custom ruff plugin or grep for ORM raw-method names with non-constant args) can flag the most common cases.

### Rationale
ORM raw-query methods are an injection vector hidden by a layer that usually protects the developer. The pattern is dangerous precisely because it looks like ORM code.

<!-- RULE END: SEC-INJ-SQL-002 -->

## Rule SEC-INJ-SQL-003

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When calling stored procedures from application code.

### Statement
Stored procedure invocations must pass arguments as bound parameters, never as part of the procedure call string.

### Violation
```python
cursor.execute(f"CALL get_user('{user_id}')")
```

### Pass
```python
cursor.callproc("get_user", [user_id])
# or
cursor.execute("CALL get_user(%s)", [user_id])
```

### Enforcement
Code review. Same SQL-concat detector flags the procedure-call form.

### Rationale
Stored procedures do not magically sanitize their inputs. Concatenation into a CALL statement is just SQL injection with a procedure name.

<!-- RULE START: SEC-INJ-SSRF-001 -->
## Rule SEC-INJ-SSRF-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_injection

### Trigger
When making outbound HTTP/network requests where the destination URL or host derives from user input (webhooks, URL previews, profile-image fetches, ingest jobs that follow a user-supplied link).

### Statement
Outbound requests with user-supplied destinations must validate the resolved host against an allowlist of permitted destinations. Block all RFC1918 ranges (10/8, 172.16/12, 192.168/16), loopback (127/8, ::1), link-local (169.254/16), cloud metadata IPs (169.254.169.254), and IPv4-mapped IPv6 forms of all of the above. Resolution and connection must use the same address to prevent DNS rebinding.

### Violation
```python
def fetch_preview(url):
    return requests.get(url, timeout=5).text
# url='http://169.254.169.254/latest/meta-data/' reads cloud creds
```

### Pass
```python
import ipaddress, socket
from urllib.parse import urlparse
ALLOWED_SCHEMES = {"http", "https"}
def fetch_preview(url):
    parsed = urlparse(url)
    if parsed.scheme not in ALLOWED_SCHEMES:
        raise ValueError("scheme not allowed")
    addr = socket.gethostbyname(parsed.hostname)
    if ipaddress.ip_address(addr).is_private or ipaddress.ip_address(addr).is_loopback:
        raise ValueError("internal address blocked")
    return requests.get(url, timeout=5).text
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_injection: regex flags `requests.get(`/`requests.post(`/`urllib.urlopen(`/`fetch(` with non-constant URL arguments. Static analysis cannot fully prove SSRF safety; the rule still requires reviewer attention.

### Rationale
SSRF is the canonical way attackers reach internal services from a public web app. Cloud metadata services (AWS IMDSv1, GCP, Azure) are the highest-impact target: they hand out credentials with no authentication. An allowlist is the only structurally safe pattern.

<!-- RULE END: SEC-INJ-SSRF-001 -->

<!-- RULE START: SEC-INJ-SSTI-001 -->
## Rule SEC-INJ-SSTI-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: false

### Trigger
When user input could reach a template-rendering engine as the template string itself, not as a value passed to a fixed template.

### Statement
Template engines (Jinja2, Twig, Handlebars, ERB) must never receive user input as the template body. User values are arguments passed to a fixed template; the template itself is always a literal or a file path under developer control.

### Violation
```python
from jinja2 import Template
template = Template(request.POST["body"])
result = template.render(name=user.name)
# body='{{config.SECRET_KEY}}' leaks secrets
```

### Pass
```python
from jinja2 import Template
TEMPLATE = Template("Hello {{ name }}, your message: {{ body }}")
result = TEMPLATE.render(name=user.name, body=request.POST["body"])
```

### Enforcement
Code review. Static analysis: flag `Template(`/`new Template(`/`compile_template(` with non-constant args.

### Rationale
Server-side template injection often gives full RCE because template engines expose object internals (Python `.__class__.__mro__`, Java reflection). Treating the template body as code, not data, removes the entire class.

<!-- RULE END: SEC-INJ-SSTI-001 -->

<!-- RULE START: SEC-INJ-XSS-001 -->
## Rule SEC-INJ-XSS-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_injection

### Trigger
When user-supplied content is rendered in HTML, including server-rendered templates, JSX, and Vue/Svelte/Blade templates.

### Statement
All user-supplied content must be rendered through the framework's automatic escaping: React JSX `{value}`, Vue/Blade/Jinja2 `{{ value }}`, Twig `{{ value }}`, Angular interpolation. Never bypass framework escaping for user data.

### Violation
```jsx
<div dangerouslySetInnerHTML={{ __html: userComment }} />
```
```php
{!! $userComment !!}
```

### Pass
```jsx
<div>{userComment}</div>
```
```php
{{ $userComment }}
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_injection: regex scan flags `dangerouslySetInnerHTML`, `v-html`, `{!! !!}`, `innerHTML =`, `outerHTML =`, `document.write(`, and `$()`.html(` with non-constant args.

### Rationale
Cross-site scripting is endemic on the public web. Every modern framework provides automatic context-aware escaping; bypassing it removes the structural defense and falls back to manual sanitization, which is almost never done correctly.

<!-- RULE END: SEC-INJ-XSS-001 -->

<!-- RULE START: SEC-INJ-XSS-002 -->
## Rule SEC-INJ-XSS-002

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: false

### Trigger
When writing JSX, Vue, Svelte, Blade, or other template code that touches any rendering API capable of inserting raw HTML.

### Statement
Never use `dangerouslySetInnerHTML`, `v-html`, `{!! !!}`, Svelte `{@html}`, or any equivalent unsafe-render API. If raw HTML is genuinely required (rich-text editor output), the content must be sanitized through a vetted library (DOMPurify, HTMLPurifier) before reaching the API.

### Violation
```jsx
function Comment({ html }) {
  return <div dangerouslySetInnerHTML={{ __html: html }} />;
}
```

### Pass
```jsx
import DOMPurify from 'dompurify';
function Comment({ html }) {
  return <div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(html) }} />;
}
// Better: render as a tree of components rather than raw HTML.
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_injection alongside SEC-INJ-XSS-001.

### Rationale
The unsafe-render APIs exist for legitimate cases (already-sanitized HTML from a server-side library), but in practice they are used for user content without sanitization. Treating their presence as a code-smell forces the author to justify why sanitization is safe at this seam.

**Mandatory**: false

### Trigger
When manipulating the DOM in vanilla JavaScript with user-controlled values.

### Statement

### Violation

```javascript
document.getElementById('msg').innerHTML = '<b>' + userMessage + '</b>';
```

```javascript
const el = document.getElementById('msg');
const bold = document.createElement('b');
bold.textContent = userMessage;
el.replaceChildren(bold);
```

Mechanically enforced by bin/run-analysis.sh::analyze_security_injection.

### Rationale
Manual DOM manipulation is rarely tested as carefully as the framework-rendered paths. The textContent/createElement form is both safe and clearer about its intent.

<!-- RULE START: SEC-RATE-API-001 -->
## Rule SEC-RATE-API-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When exposing public API endpoints (REST, GraphQL, RPC) to internet traffic.

### Statement
Public API endpoints must be rate-limited per client identity (API key, authenticated user, or source IP for unauthenticated). The limit applies at the application or gateway layer, not just at the network layer.

### Violation
```python
@app.get('/api/search')
def search(q):
    return expensive_search(q)
# Unlimited; one bot can hammer the endpoint into a degraded state.
```

### Pass
```python
@app.get('/api/search')
@limiter.limit('60 per minute', key_func=client_identity)
def search(q):
    return expensive_search(q)
```

### Enforcement
Middleware config review. API gateways (Kong, Tyk, AWS API Gateway) provide native support.

### Rationale
Without rate limits, a single client can exhaust capacity for everyone else. Per-identity limits provide fairness and a per-tenant denial-of-service floor.

<!-- RULE END: SEC-RATE-API-001 -->

<!-- RULE START: SEC-RATE-BATCH-001 -->
## Rule SEC-RATE-BATCH-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing bulk/batch endpoints that accept an array of operations (bulk create, bulk update, multi-delete).

### Statement
Batch endpoints cap the number of items accepted per request AND apply a per-batch rate limit. Unbounded batch APIs let a single request consume disproportionate resources.

### Violation
```python
@app.post('/api/users/bulk')
def bulk_create(users: list[UserCreate]):
    return [User.create(**u.dict()) for u in users]
```

### Pass
```python
MAX_BATCH = 100
@app.post('/api/users/bulk')
@limiter.limit('10 per minute')
def bulk_create(users: list[UserCreate]):
    if len(users) > MAX_BATCH:
        abort(400)
    return [User.create(**u.dict()) for u in users]
```

### Enforcement
Code review.

### Rationale
Batch endpoints amplify request cost without amplifying client cost; capping batch size keeps cost ratio bounded.

## Rule SEC-RATE-LOGIN-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing authentication endpoints (login, password reset, MFA verify, token exchange).

### Statement
Authentication endpoints carry their own rate limit, distinct from the general API limit. Login limits are per account AND per IP (per-account stops credential stuffing of a known user; per-IP stops bulk username enumeration).

### Violation
```python
@app.post('/login')
def login():
    ...
# Falls under the general /api 60-per-minute limit -- enough room for
# 60 credential-stuffing attempts per minute per IP.
```

### Pass
```python
@app.post('/login')
@limiter.limit('5 per 15 minutes', key_func=lambda: request.json['email'])
@limiter.limit('20 per 15 minutes', key_func=get_remote_address)
def login():
    ...
```

### Enforcement
Code review.

### Rationale
Login endpoints are the highest-value attack surface; a per-account limit defeats stuffing of a target user, and a per-IP limit defeats horizontal sweeps.

<!-- RULE START: SEC-RATE-QUERY-001 -->
## Rule SEC-RATE-QUERY-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing endpoints that trigger database queries based on user input (list, search, filter, export, GraphQL collection fields).

### Statement
Database queries triggered by user input must apply pagination with a maximum page size, and unbounded result sets are not returned. The default page size is small; the max is bounded.

### Violation
```python
@app.get('/api/orders')
def list_orders():
    return Order.query.all()  # could be 10 million rows
```

### Pass
```python
@app.get('/api/orders')
def list_orders():
    page = int(request.args.get('page', 1))
    per_page = min(int(request.args.get('per_page', 25)), 200)
    return Order.query.paginate(page=page, per_page=per_page).items
```

### Enforcement
Code review. GraphQL: depth-limit / cost-analysis plugins.

### Rationale
An unbounded query is a database DoS waiting to happen: one user with a wide filter holds a worker and a connection for minutes. Bounded pagination caps the worst case.

<!-- RULE END: SEC-RATE-QUERY-001 -->

<!-- RULE START: SEC-RATE-UPLOAD-001 -->
## Rule SEC-RATE-UPLOAD-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing file upload endpoints.

### Statement
File upload endpoints enforce explicit size limits at the application layer: max bytes per file AND max bytes per request body. The application rejects oversized requests before reading them entirely into memory.

### Violation
```python
@app.post('/upload')
def upload():
    f = request.files['file']
    f.save('/tmp/...')  # whatever the client sent, however large
```

### Pass
```python
app.config['MAX_CONTENT_LENGTH'] = 10 * 1024 * 1024  # 10 MB
@app.post('/upload')
def upload():
    f = request.files['file']
    if len(f.read()) > 5 * 1024 * 1024:  # 5 MB per file
        abort(413)
    ...
```

### Enforcement
Framework config review (Flask MAX_CONTENT_LENGTH, Django DATA_UPLOAD_MAX_MEMORY_SIZE, nginx client_max_body_size).

### Rationale
Unbounded upload endpoints are a direct DoS vector: a single 4GB POST can exhaust memory or fill disk. Application-layer caps prevent the request from ever reaching that state.

<!-- RULE END: SEC-RATE-UPLOAD-001 -->

<!-- RULE START: SEC-VAL-ALLOW-001 -->
## Rule SEC-VAL-ALLOW-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When validating inputs whose valid set is enumerable (countries, statuses, roles, file types, sort fields, redirect targets).

### Statement
Validation uses an allowlist (explicit set of permitted values) whenever the valid set is enumerable. Blocklists (reject these characters/values) are forbidden where allowlists are possible: blocklists are routinely incomplete and silently fail open on unanticipated inputs.

### Violation
```python
if '<script>' in user_html or 'javascript:' in user_html:
    abort(400)
# Blocklist; bypassed by '<svg onload=...>'.
```

### Pass
```python
ALLOWED_TAGS = {'p', 'br', 'em', 'strong', 'a', 'ul', 'li'}
sanitized = bleach.clean(user_html, tags=ALLOWED_TAGS, attributes={'a': ['href']})
```

### Enforcement
Code review. Look for negative-pattern checks (blocklists) on fields with an enumerable valid set.

### Rationale
Allowlists fail closed: anything not on the list is rejected. Blocklists fail open: anything the author did not anticipate is allowed. The asymmetry favors allowlists structurally.

<!-- RULE END: SEC-VAL-ALLOW-001 -->

<!-- RULE START: SEC-VAL-ENCODING-001 -->
## Rule SEC-VAL-ENCODING-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When validating any input that may have been URL-encoded, HTML-entity-encoded, or unicode-normalized differently from canonical form.

### Statement
Input is decoded to canonical form (single decode, NFC unicode normalization) before validation. Double-decoding, accepting mixed encodings, or validating before decoding allows bypass through alternative encodings.

### Violation
```python
if '..' in path:  # rejects '..' but accepts '..%2f' or '%2e%2e/'
    abort(400)
open(unquote(path))  # decoded after the check
```

### Pass
```python
canonical = unicodedata.normalize('NFC', unquote(path))
if '..' in canonical:
    abort(400)
open(canonical)
```

### Enforcement
Code review. Look for validation followed by decoding/normalization rather than the reverse.

### Rationale
Encoding bypass is one of the oldest WAF-evasion tricks. Canonical-form validation removes the trick by collapsing all encodings to a single representation before the check runs.

<!-- RULE START: SEC-VAL-FILE-001 -->
## Rule SEC-VAL-FILE-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When accepting file uploads from users (avatars, attachments, documents, media).

### Statement
File uploads must be validated by content (magic bytes / libmagic / file-type sniffing), not just by extension or client-supplied MIME type. Files are stored outside the web root, served through an application handler that sets safe Content-Type and Content-Disposition headers. Filename is sanitized or replaced with a UUID.

### Violation
```python
@app.route('/upload', methods=['POST'])
def upload():
    f = request.files['file']
    if f.filename.endswith('.pdf'):
        f.save(f'static/uploads/{f.filename}')  # web-root, attacker-controlled name
```

### Pass
```python
import magic, uuid
@app.route('/upload', methods=['POST'])
def upload():
    f = request.files['file']
    mime = magic.from_buffer(f.read(2048), mime=True)
    if mime not in {'application/pdf', 'image/png', 'image/jpeg'}:
        abort(400)
    f.seek(0)
    storage_name = f'{uuid.uuid4()}.bin'
    f.save(f'/var/app-uploads/{storage_name}')
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_auth_authz: regex flags `request.files[...].save(`, `$request->file(...)->store(`, `multer({ dest: ... })`, and similar patterns to direct manual review.

### Rationale
Extension-only and MIME-only checks are trivially spoofed. Content sniffing closes the spoof; out-of-web-root storage closes the direct-execute vector; sanitized filenames close the path-traversal vector. All three are required.

## Rule SEC-VAL-LENGTH-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When defining schemas or accepting string-typed fields from user input.

### Statement
All string inputs must have an explicit maximum length constraint. Unbounded text fields (free-form descriptions, comments, names) without a max_length are violations. The default for any new string field is bounded.

### Violation
```python
class Comment(BaseModel):
    body: str  # no length cap
```

### Pass
```python
class Comment(BaseModel):
    body: constr(min_length=1, max_length=10_000)
```

### Enforcement
Schema review. Look for `str`, `Optional[str]`, `TEXT`/`VARCHAR` columns without length constraints.

### Rationale
Unbounded input fields are a denial-of-service vector (gigabyte payloads) and a database-bloat vector. Length caps are zero-cost and prevent both.

## Rule SEC-VAL-RANGE-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When accepting numeric inputs (counts, IDs, prices, percentages, durations, sizes).

### Statement
Numeric inputs must be validated against the domain-appropriate range. Negative quantities, integer overflow values, and out-of-domain numbers (e.g., percentage > 100, year < 1900) are rejected at validation time.

```python
class Order(BaseModel):
```

```python
class Order(BaseModel):
```

Schema review.

### Rationale

<!-- RULE START: SEC-VAL-REGEX-001 -->
## Rule SEC-VAL-REGEX-001

**Domain**: security
**Severity**: Medium
**Scope**: Component
**Mandatory**: false

### Trigger
When accepting regex patterns from end users (search, filter, advanced query, admin UI).

### Statement
User-supplied regular expressions must be rejected outright or executed under a non-backtracking engine (RE2, Hyperscan) with a time bound. The default Python `re`, Node `RegExp`, PHP `preg_match`, and Java `Pattern` engines are exposed to ReDoS.

### Violation
```python
pattern = request.GET.get('pattern')
matches = [s for s in items if re.search(pattern, s.name)]
```

### Pass
```python
import re2  # google-re2 binding
pattern = request.GET.get('pattern')
try:
    compiled = re2.compile(pattern, max_mem=8*1024*1024)
except re2.error:
    abort(400)
matches = [s for s in items if compiled.search(s.name)]
```

### Enforcement
Code review. Look for `re.compile(user_input)` or equivalent patterns.

### Rationale
A catastrophic-backtracking regex can hang a worker for minutes on a single short input. Without a non-backtracking engine or a timeout, this is a single-request denial of service.

<!-- RULE START: SEC-VAL-SERVER-001 -->
## Rule SEC-VAL-SERVER-001

**Domain**: security
**Severity**: Critical
**Scope**: Component
**Mandatory**: true
**Mechanical_Enforcement_Path**: bin/run-analysis.sh::analyze_security_auth_authz

### Trigger
When accepting any data from a client (HTTP request, form, file upload, websocket message, mobile-app payload, third-party webhook).

### Statement
All input validation is performed server-side, before the data reaches business logic. Client-side validation exists for UX only and is never trusted. Trusting a custom HTTP header or a hidden form field as 'pre-validated' is a violation.

### Violation
```javascript
// Frontend
if (form.email.match(EMAIL_REGEX)) { submit(form); }
// Backend
app.post('/signup', (req, res) => {
  db.users.create(req.body);  // assumes frontend validated
});
```

### Pass
```javascript
// Backend
app.post('/signup', (req, res) => {
  const { error, value } = signupSchema.validate(req.body);
  if (error) return res.status(400).json({ error });
  db.users.create(value);
});
```

### Enforcement
Mechanically enforced by bin/run-analysis.sh::analyze_security_auth_authz: presence check flags handler functions that read req.body / request.json directly without an intervening schema-validation call. Detection is conservative; code review covers the rest.

### Rationale
Every byte from the client is hostile until validated. Server-side validation is the only trust boundary that matters; client validation is an accessibility helper, not a security control.

<!-- RULE START: SEC-VAL-TYPE-001 -->
## Rule SEC-VAL-TYPE-001

**Domain**: security
**Severity**: High
**Scope**: Component
**Mandatory**: false

### Trigger
When implementing any endpoint that accepts a JSON or form body.

### Statement
Request payloads must be validated against a typed schema (Pydantic, Marshmallow, Zod, Yup, JSON Schema, Joi, class-validator, Bean Validation) before business logic runs. Hand-rolled `if 'field' in request.json` checks are insufficient.

### Violation
```python
@app.route('/api/users', methods=['POST'])
def create_user():
    data = request.json
    if 'email' in data:
        User.create(email=data['email'])
```

### Pass
```python
class UserCreate(BaseModel):
    email: EmailStr
    name: constr(max_length=120)

@app.route('/api/users', methods=['POST'])
def create_user():
    payload = UserCreate(**request.json)
    User.create(**payload.dict())
```

### Enforcement
Code review. Look for handler functions that read request.json/request.body without an immediate schema constructor or validate() call.

### Rationale
Schemas turn validation from a manually-maintained pile of `if` statements into a single declarative source of truth that also generates docs, types, and OpenAPI. The structural win is enormous.

<!-- RULE END: SEC-VAL-TYPE-001 -->
