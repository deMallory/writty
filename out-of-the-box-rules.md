Writ Public Rulebook — Out-of-the-Box Rule Inventory
Every rule listed here must pass the enforceability test: an AI agent can make
a binary pass/fail decision from the rule's trigger + statement + violation
fields without human interpretation.
Rules marked mandatory are loaded every turn via /always-on and never
ranked. Rules marked retrieved are indexed by BM25 + vector and surfaced
only when relevant. The split matters: mandatory rules are safety invariants
that cannot be ranked away; retrieved rules are guidance that should appear
when the task matches.
Total: 220 rules across 12 domains.

1. Security (CRITICAL — the most vital domain)
This is the domain where mandatory vs retrieved matters most. A missed
security rule is a shipped vulnerability. The mandatory set covers the
universal invariants; the retrieved set covers context-specific guidance.

Notes for Implementation
Severity assignment logic

Critical: A violation creates an exploitable vulnerability, data loss, or system failure
in production. No exceptions.
High: A violation causes bugs, maintenance debt, or security weakness that compounds
over time. Exceptions require documented justification.
Medium: A violation degrades code quality or developer experience. Exceptions
are acceptable with team agreement.
Low: A violation is a style or hygiene issue. Advisory only.

Mandatory selection criteria
A rule is mandatory (always-on, never ranked) when ALL of these are true:

Severity is critical
A violation is exploitable or causes data loss in production
The rule is universal (applies regardless of language, framework, or domain)
An AI agent can mechanically detect violations from code inspection

Language-specific extensions (included as examples)
The 220 rules above are language-agnostic. Writ ships with the maintainer's own
language-specific bundles as working examples of how to extend the rulebook for
your stack:

Python: PY-* rules (type hints, async patterns, import conventions, virtual environments)
JavaScript/TypeScript: JS-/TS- rules (strict mode, Promise handling, null safety, module patterns)
Rust: RS-* rules (ownership patterns, unsafe blocks, error handling with Result)
Go: GO-* rules (error returns, goroutine lifecycle, context propagation)
SQL: SQL-* rules (index usage, transaction isolation, migration patterns)

These are retrieved, not mandatory, and scoped to their language domain. Users
can add their own language bundles following the same pattern.
